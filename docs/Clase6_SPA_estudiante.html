01 · PUNTO DE PARTIDA

🔁 Lo que ya sabes de la clase pasada

Todo lo de hoy se para sobre lo que viste en la Clase 5. Antes de avanzar, refresquemos lo esencial, porque lo vas a usar de inmediato.

Encapsulamiento

Encapsular es ocultar el estado interno de un objeto y exponerlo solo a través de métodos controlados.

En tu cuenta bancaria, el saldo era private: nadie de afuera lo tocaba directo, solo depositar() y retirar(), que validaban antes de modificar.

Idea clave

La encapsulación no es solo esconder; es proteger con reglas de negocio.

¿Por qué usamos modificadores de acceso?

Porque deciden quién puede tocar qué. No es capricho: son lo que hace posible la encapsulación.

public → cualquiera, desde cualquier clase.

private → solo la propia clase.

protected → la clase y sus hijas. Hoy lo estrenas.

(nada) → solo el mismo paquete (default).

La palabra que quedó pendiente: sobreescritura

Al final de la Clase 5 quedó una promesa. Guarda esta diferencia, porque hoy la vives:

Sobrecarga (Clase 5): mismo nombre de método, distintos parámetros, en la misma clase.

Sobreescritura (hoy): una clase hija redefine un método que heredó del padre. Es cosa de herencia.

También viste toString(), que define cómo se ve un objeto en texto. Hoy lo reutilizas.

02 · REUTILIZAR

🧬 Herencia con extends

Imagina que construyes una plataforma tipo tienda online. Tienes clientes que compran, vendedores que publican y administradores que gestionan. Los tres son usuarios: los tres tienen nombre y correo, los tres inician sesión.

¿Creas tres clases totalmente separadas y copias el nombre y el correo en cada una?
▸ Ver explicación
Se puede, pero estarías copiando y pegando los mismos atributos y métodos en tres lugares. Si mañana el correo cambia de reglas, tendrías que corregirlo tres veces. Cuando varias clases comparten lo mismo, ese "lo mismo" debería vivir en un solo lugar. Esa es la puerta de entrada a la herencia.

Herencia: mecanismo por el cual una clase (hija) adquiere los atributos y métodos de otra (padre) y puede además agregar los suyos. Se declara con extends.

Un smartphone es un teléfono: hereda "hacer llamadas" y encima agrega cámara, apps, internet. No reinventa el llamar; lo hereda y suma lo suyo.

Regla mental

Herencia = relación "es un / es una". Un Cliente es un Usuario.

Ejemplo aislado: Vehiculo y Moto

El padre Vehiculo define marca, velocidad y acelerar(). La hija Moto hereda todo eso con extends y agrega lo suyo. Fíjate en super(marca): la moto le pasa la marca al constructor del padre.

Vehiculo.java · crear archivo nuevo 
public class Vehiculo {
    protected String marca;
    protected int velocidad;

    public Vehiculo(String marca) {
        this.marca = marca;
        this.velocidad = 0;
    }

    public void acelerar() {
        velocidad += 10;
        System.out.println(marca + " acelera a " + velocidad + " km/h");
    }
}
Ojo al protected

Aquí estrenas el modificador que quedó pendiente. marca es protected para que la hija pueda usarlo, pero siga oculto para el resto del mundo. Ni tan abierto (public) ni tan cerrado (private).

Moto.java · crear archivo nuevo 
public class Moto extends Vehiculo {
    public Moto(String marca) {
        super(marca);
    }

    public void hacerCaballito() {
        System.out.println(marca + " hace un caballito");
    }
}
AppVehiculo.java · con el main 
public class AppVehiculo {
    public static void main(String[] args) {
        Moto moto = new Moto("Yamaha");
        moto.acelerar();
        moto.hacerCaballito();
    }
}

Imprime:

Yamaha acelera a 10 km/h
Yamaha hace un caballito

Idea clave

moto.acelerar() funciona sin escribirlo en Moto. Se heredó de Vehiculo. Escribes una vez, reutilizas siempre.

Así nace nuestro hilo: Usuario y Cliente

Creamos el padre Usuario con lo común y una hija Cliente que solo agrega lo suyo (puntos). Nota que Cliente no repite nombre ni email: los hereda.

Usuario.java · crear archivo nuevo 
public class Usuario {
    protected String nombre;
    protected String email;

    public Usuario(String nombre, String email) {
        this.nombre = nombre;
        this.email = email;
    }

    public String getNombre() {
        return nombre;
    }

    public String getEmail() {
        return email;
    }
}
Cliente.java · crear archivo nuevo 
public class Cliente extends Usuario {
    private int puntos;

    public Cliente(String nombre, String email) {
        super(nombre, email);
        this.puntos = 0;
    }
}
03 · REDEFINIR

✍️ Sobreescritura con @Override

Ya tienes Cliente, Vendedor y Administrador heredando de Usuario. Pero el panel de inicio de cada uno es distinto: el cliente ve sus compras, el admin ve reportes, el vendedor ve sus productos.

Si el método panelInicio() vive en el padre, ¿cómo hace cada hijo para mostrar su panel y no uno genérico?
▸ Ver explicación
La tentación es un if gigante que pregunte el tipo, pero eso es justo lo que la POO viene a eliminar. La solución elegante es que cada hija redefina el método a su manera. Eso es sobreescritura.

Sobreescritura (@Override): una clase hija redefine un método que heredó del padre, dándole su propia implementación. La firma es la misma; cambia el cuerpo.

Todos los empleados de una empresa "saludan al cliente" (método heredado). Pero el de ventas saluda distinto al de soporte. Mismo deber, forma distinta.

La anotación @Override

Le avisa al compilador "esto redefine algo del padre". Si te equivocas en el nombre, te avisa con un error en vez de crear un método nuevo por accidente. Es una red de seguridad.

Ejemplo aislado: Animal y Gato

El padre Animal tiene un hacerSonido() genérico. Gato lo sobreescribe para devolver "Miau".

Animal.java · crear archivo nuevo 
public class Animal {

    public Animal(String nombre) {
        this.nombre = nombre;
    }

    public String hacerSonido() {
        return "Sonido generico";
    }
}
Gato.java · crear archivo nuevo 
public class Gato extends Animal {
    public Gato(String nombre) {
        super(nombre);
    }

    @Override
    public String hacerSonido() {

}

04 · UNA LISTA, MUCHAS FORMAS

🎭 Polimorfismo: el premio de sobreescribir

El admin quiere ver, en pantalla, el panel de todos los usuarios de la plataforma. Son cientos, mezclados: clientes, vendedores, admins.

¿Los recorres con listas separadas y un if por tipo, o puedes meterlos todos en una sola lista y que cada uno muestre su panel solo?
▸ Ver explicación
Puedes meterlos todos juntos en una lista de tipo Usuario y recorrerla una sola vez. Cada objeto, aunque lo trates como Usuario, responde según lo que realmente es. Eso es polimorfismo.

Polimorfismo: capacidad de tratar objetos de distintas clases hijas a través del tipo del padre, de modo que cada uno responde con su propia versión sobreescrita del método. Un mismo llamado, muchas formas de responder.

Un control remoto con un botón "Play". Sirve para el TV, el equipo y el DVD. Un solo botón (mismo llamado), y cada aparato reacciona a su manera.

Cómo se ve en código

Guardamos hijas distintas en un arreglo del tipo del padre (Usuario[]). Al recorrerlo y llamar panelInicio(), Java ejecuta automáticamente la versión de cada hija. En este punto todas tus clases viven juntas en la misma carpeta; más adelante las organizamos en paquetes.

PruebaPolimorfismo.java · con el main 
public class PruebaPolimorfismo {
    public static void main(String[] args) {
        Usuario[] usuarios = {
            new Cliente("Ana", "ana@mail.com"),
            new Administrador("Luis", "luis@mail.com"),
            new Vendedor("Marta", "marta@mail.com")
        };

        for (Usuario u : usuarios) {
            System.out.println(u.getNombre() + " -> " + u.panelInicio());
        }
    }
}

Imprime:

Ana -> Panel de Cliente: ver productos y mis compras
Luis -> Panel de Admin: gestionar usuarios y reportes
Marta -> Panel de Vendedor: publicar y gestionar mis productos

Idea clave

La variable es Usuario, pero cada objeto recuerda quién es. El bucle nunca pregunta "¿eres cliente? ¿eres admin?", y aun así cada uno responde distinto. Agregar un nuevo tipo de usuario mañana no te obliga a tocar este bucle. Eso es código que escala.

05 · PROHIBIR LO IMPOSIBLE

🚫 Clases abstractas

En tu plataforma, ¿tiene sentido que exista un usuario que sea solo Usuario, sin ser ni cliente, ni vendedor, ni admin? ¿Un usuario "genérico" suelto?
▸ Ver explicación
No. En el negocio, todo usuario es de algún tipo concreto. Un Usuario genérico no debería poder crearse: es una plantilla incompleta, no una entidad real. Java tiene una herramienta para decir "esta clase existe para ser heredada, pero no para instanciarse": la clase abstracta.

Clase abstracta: clase marcada con abstract que no se puede instanciar con new. Sirve como base. Puede tener métodos abstractos (declarados sin cuerpo) que obligan a las hijas a implementarlos.

"Vehículo" en abstracto no existe en la calle: no compras "un vehículo genérico". Compras un carro, una moto, un bus. Lo que rueda es siempre un tipo concreto.

El método abstracto: de dónde sale y por qué se ve raro

Sigamos la pista de panelInicio(), porque aquí se transforma:

En la sección anterior nació con cuerpo en el padre: tenía llaves { } y un return.

Ahora, al volver Usuario abstracta, ese "panel genérico" no significa nada. Le quitamos el cuerpo y lo dejamos como contrato:

método abstracto 
public abstract String panelInicio();

Un método abstract no tiene cuerpo: no lleva llaves, solo la firma y un ;. Eso es lo que se ve raro la primera vez. Su mensaje es: "yo, el padre, declaro que este método existe, pero NO lo implemento; obligo a cada hija a definirlo".

Sobre public abstract

La palabra abstract es obligatoria (es la que dice "sin cuerpo"). El public es solo visibilidad; podría ser protected abstract. Lo único prohibido es private abstract: sería una contradicción (te obligo a redefinirlo, pero no dejo que nadie lo vea).

Ejemplo aislado: Figura

No existe "una figura" a secas; existen círculos, rectángulos. Figura es abstracta y su calcularArea() es abstracto: cada figura lo calcula distinto, pero todas están obligadas a tenerlo. Lo retomamos completo al final.

Figura.java · crear archivo nuevo 
public abstract class Figura {
    public abstract double calcularArea();
}
Error a propósito, para verlo en vivo

Con Usuario ya abstract, si en un main intentas new Usuario("X", "x@mail.com");, el compilador no deja y muestra: "Usuario is abstract; cannot be instantiated". Ese error es la regla de negocio funcionando: nadie crea un usuario "sin tipo".

Así va el hilo: Usuario abstracta

Convertimos Usuario en abstracta y su panelInicio() en método abstracto. Ya no damos panel genérico: obligamos a cada hija a definir el suyo. También traemos aquí el toString() de la Clase 5.

Usuario.java · así va la clase 
public abstract class Usuario {
    protected String nombre;
    protected String email;

    public Usuario(String nombre, String email) {
        this.nombre = nombre;
        this.email = email;
    }

    public String getNombre() {
        return nombre;
    }

    public String getEmail() {
        return email;
    }

    public abstract String panelInicio();

    @Override
    public String toString() {
        return nombre + " <" + email + ">";
    }
}
Idea clave

panelInicio() ya no tiene cuerpo en el padre. Si una hija olvida implementarlo, no compila. La clase abstracta garantiza que ninguna hija quede incompleta.

06 · QUÉ PUEDE HACER

📜 Interfaces

Quieres enviar notificaciones. Pero ojo: notifican los clientes (por email) y los vendedores (por SMS), pero los administradores no. "Poder ser notificado" no es un tipo de usuario: es una capacidad que solo algunos tienen.

¿La metes en la clase Usuario de la que todos heredan?
▸ Ver explicación
Si la pones en Usuario, el admin también la hereda y no debería. "Ser notificable" no describe qué es el objeto (eso lo hace la herencia), sino qué puede hacer. Para capacidades transversales que solo algunas clases tienen, Java usa interfaces.

Interfaz: un contrato que declara métodos que una clase se compromete a implementar, sin decir cómo. Se adopta con implements. Una clase hereda de una sola clase, pero puede implementar varias interfaces.

"Ser mamífero" es qué eres (herencia): naces siéndolo. "Saber nadar" es qué puedes hacer (interfaz): lo cumplen el pez, el humano y el perro, que no son de la misma familia.

La distinción clave del día

Herencia / clase abstracta responde a "qué ES" el objeto · palabra extends · solo una padre · un Cliente es un Usuario.

Interfaz responde a "qué PUEDE HACER" · palabra implements · varias a la vez · un Cliente puede ser Notificable.

Ejemplo aislado: MetodoPago

Distintas formas de pago (tarjeta, PSE) no son "familia" entre sí, pero todas se comprometen a procesarPago(). Lo retomamos completo al final.

MetodoPago.java · crear archivo nuevo 
public interface MetodoPago {
    String procesarPago(double monto);
}
Error a propósito, para verlo en vivo

En Cliente, agrega implements Notificable pero no escribas todavía el método. El error salta en la línea de la clase: "Cliente is not abstract and does not override abstract method recibirNotificacion(String) in Notificable". Significa: firmaste el contrato pero no lo cumpliste.

Así va el hilo: Notificable

Creamos la interfaz. Luego solo Cliente y Vendedor la implementan, cada uno a su manera; Administrador no, porque no recibe notificaciones.

Notificable.java · crear archivo nuevo 
public interface Notificable {
    String recibirNotificacion(String mensaje);
}
Cliente.java · así queda 
public class Cliente extends Usuario implements Notificable {
    private int puntos;

    public Cliente(String nombre, String email) {
        super(nombre, email);
        this.puntos = 0;
    }

    @Override
    public String panelInicio() {
        return "Panel de Cliente: ver productos y mis compras";
    }

    @Override
    public String recibirNotificacion(String mensaje) {
        return "Email a " + email + ": " + mensaje;
    }
}
Vendedor.java · así queda 
public class Vendedor extends Usuario implements Notificable {
    public Vendedor(String nombre, String email) {
        super(nombre, email);
    }

    @Override
    public String panelInicio() {
        return "Panel de Vendedor: publicar y gestionar mis productos";
    }

    @Override
    public String recibirNotificacion(String mensaje) {
        return "SMS al vendedor " + nombre + ": " + mensaje;
    }
}
Fíjate en la línea

class Cliente extends Usuario implements Notificable. Hereda de una clase Y cumple un contrato, a la vez. Administrador se queda solo con extends Usuario.

07 · ORDENAR EL PROYECTO

🏛️ Arquitectura por capas (limpia inicial)

Ya tienes Usuario, tres hijas, una interfaz y un main que lo usa todo.

Si dejas todos los archivos revueltos en una sola carpeta, ¿qué pasa cuando el proyecto tenga 200 clases? ¿Dónde vive la lógica del negocio, dónde lo que ve el usuario?
▸ Ver explicación
Buscar por nombre no escala. Los proyectos serios separan responsabilidades en capas, cada una en su paquete. Así el negocio no se mezcla con la pantalla ni con los detalles técnicos. Ese es el corazón de la arquitectura limpia.

Arquitectura por capas: organizar el código según responsabilidades, cada una en su paquete. Usamos los mismos nombres de Spring Boot, para que cuando llegues allá ya te suenen.

Un restaurante: el model es la cocina (el saber hacer), el service es el mesero (lleva y trae), el controller es el comedor (donde el cliente pide). La cocina no necesita saber cómo están decoradas las mesas.

model → las entidades y reglas del negocio (Usuario, hijas, Notificable). El corazón.

service → orquesta el negocio (procesar la lista de usuarios).

controller → lo que interactúa con el usuario (hoy el main; en Spring, los endpoints).

Falta una capa que verás pronto

repository, la que habla con la base de datos. Hoy no aparece porque todavía no guardamos datos en ninguna parte. La estrenas en la próxima clase.

La regla de oro

Las dependencias apuntan hacia el model, nunca al revés. El controller conoce al service; el service conoce al model. El model no conoce a nadie: no sabe que existe una pantalla ni una base de datos.

Otros nombres, misma idea

Quizás veas por ahí dominio / servicio / presentacion: son los nombres conceptuales de Clean Architecture y significan exactamente lo mismo. La equivalencia es directa: model ≈ dominio, service ≈ servicio, controller ≈ presentación.

Cómo queda organizado

estructura de carpetas 
plataforma/
├── model/
│   ├── Usuario.java
│   ├── Notificable.java
│   ├── Cliente.java
│   ├── Administrador.java
│   └── Vendedor.java

Dos cosas nuevas al separar en paquetes

package: la primera línea de cada archivo dice a qué paquete pertenece. import: cuando una clase usa otra de otro paquete, la importa arriba. Las del mismo paquete no se importan entre sí.

La capa service

Recibe el arreglo de usuarios y los procesa. mostrarPaneles usa el polimorfismo. En notificarATodos aparece instanceof: pregunta "¿este usuario cumple el contrato Notificable?" y solo notifica a los que sí. La línea (Notificable) u es un cast: "trátalo ahora como Notificable".

service/PlataformaService.java 
package plataforma.service;

import plataforma.model.Usuario;
import plataforma.model.Notificable;

public class PlataformaService {

    public void mostrarPaneles(Usuario[] usuarios) {
        for (Usuario u : usuarios) {
            System.out.println(u.getNombre() + " -> " + u.panelInicio());
        }
    }

    public void notificarATodos(Usuario[] usuarios, String mensaje) {
        for (Usuario u : usuarios) {
            if (u instanceof Notificable) {
                Notificable n = (Notificable) u;
                System.out.println(n.recibirNotificacion(mensaje));
            }
        }
    }
}

La capa controller (junta todo)

El main arma la lista mezclada y le pide todo al service. No calcula nada por su cuenta: solo arma los datos y delega. Aquí se juntan todos los conceptos de la clase.

controller/App.java · con el main 
package plataforma.controller;

import plataforma.model.Usuario;
import plataforma.model.Cliente;
import plataforma.model.Administrador;
import plataforma.model.Vendedor;
import plataforma.service.PlataformaService;

public class App {
    public static void main(String[] args) {
        Usuario[] usuarios = {
            new Cliente("Ana", "ana@mail.com"),
            new Administrador("Luis", "luis@mail.com"),
            new Vendedor("Marta", "marta@mail.com")
        };

        PlataformaService servicio = new PlataformaService();

        System.out.println("--- Paneles ---");
        servicio.mostrarPaneles(usuarios);

        System.out.println("--- Notificaciones ---");
        servicio.notificarATodos(usuarios, "Oferta del dia");
    }
}

Imprime:

--- Paneles ---
Ana -> Panel de Cliente: ver productos y mis compras
Luis -> Panel de Admin: gestionar usuarios y reportes
Marta -> Panel de Vendedor: publicar y gestionar mis productos
--- Notificaciones ---
Email a ana@mail.com: Oferta del dia
SMS al vendedor Marta: Oferta del dia

Lectura de negocio

Los tres muestran su panel (polimorfismo). En las notificaciones, Luis el admin no aparece: no implementa Notificable. Cliente notifica por email, vendedor por SMS. Todo sin un solo if preguntando "¿eres cliente? ¿eres vendedor?".

Crear proyecto y correr — IntelliJ / VS Code

Crear paquete: en IntelliJ, clic derecho sobre src → New → Package (plataforma.model). En VS Code, creas las carpetas y mueves cada .java. Ejecutar: en ambos, botón ▶️ Run sobre App, o por terminal javac + java. El Java es idéntico en los dos; solo cambian los atajos.

08 · CONSOLIDAR (ABSTRACTA)

🔷 Cierre: Figuras geométricas

Este ejemplo refuerza clase abstracta + polimorfismo por herencia, desde otro ángulo. Ya viste Figura y Circulo. Agregamos Rectangulo y las recorremos polimórficamente: mismo calcularArea(), resultados distintos.

Circulo.java · crear archivo nuevo 
public class Circulo extends Figura {
    private double radio;

    public Circulo(double radio) {
        this.radio = radio;
    }

    @Override
    public double calcularArea() {
        return Math.PI * radio * radio;
    }
}
Rectangulo.java · crear archivo nuevo 
public class Rectangulo extends Figura {
    private double base;
    private double altura;

    public Rectangulo(double base, double altura) {
        this.base = base;
        this.altura = altura;
    }

    @Override
    public double calcularArea() {
        return base * altura;
    }
}
AppFiguras.java · con el main 
public class AppFiguras {
    public static void main(String[] args) {
        Figura[] figuras = {
            new Circulo(5),
            new Rectangulo(4, 6)
        };
        for (Figura f : figuras) {
            System.out.printf("Area: %.2f%n", f.calcularArea());
        }
    }
}

Imprime:

Area: 78.54
Area: 24.00

09 · CONSOLIDAR (INTERFAZ)

🔶 Cierre: Medios de pago

Este ejemplo refuerza interfaz + polimorfismo por contrato. Ya viste MetodoPago y PagoTarjeta. Agregamos PagoPSE y los tratamos a todos como MetodoPago: mismo contrato, cada uno lo cumple a su manera.

PagoTarjeta.java · crear archivo nuevo 
public class PagoTarjeta implements MetodoPago {
    @Override
    public String procesarPago(double monto) {
        return "Cobrando $" + monto + " a la tarjeta de credito";
    }
}
PagoPSE.java · crear archivo nuevo 
public class PagoPSE implements MetodoPago {
    @Override
    public String procesarPago(double monto) {
        return "Redirigiendo al banco para pagar $" + monto + " por PSE";

}
AppPagos.java · con el main 
public class AppPagos {
    public static void main(String[] args) {
        MetodoPago[] metodos = {
            new PagoTarjeta(),

El contraste, lado a lado

10 · PRÁCTICA

🤖 Refactorización con Cursor AI

La IA propone; tú auditas con criterio de arquitecto. Ese criterio es justo lo que construiste hoy.

Ejercicio guiado

Pídele a Cursor AI que agregue un nuevo tipo de usuario, UsuarioInvitado, que hereda de Usuario pero no implementa Notificable (un invitado no recibe notificaciones). Luego audita con criterio:

¿Respetó que Usuario es abstracta?

¿Sobreescribió panelInicio() con @Override?

¿Puso la clase en el paquete model correcto?

¿Rompió la regla de las capas (metió lógica de controller dentro del model)?

Reto de refactor

Pídele a Cursor que extraiga la lógica de "cómo se envía" la notificación (email vs SMS) hacia algo más limpio. Discute: ¿esto se parece a una nueva interfaz? Sin implementarlo aún, solo detectar la oportunidad. Ese olfato es señal de que la arquitectura ya se te está volviendo natural.

11 · RESUMEN

✅ Lo que te llevas hoy

Herencia (extends): una hija reutiliza lo del padre. Relación "es un".

protected: abre los atributos del padre a las hijas, sin abrirlos al mundo.

Sobreescritura (@Override): la hija redefine un método heredado. Mismo deber, forma distinta.

Polimorfismo: tratar hijas por el tipo del padre; cada una responde a su manera. Adiós al if gigante.

Clase abstracta: base que no se instancia y obliga a las hijas a completar métodos.

Interfaz (implements): contrato de capacidades. Responde "qué puede hacer", no "qué es". Puedes implementar varias.

Arquitectura por capas: model, service, controller (y repository viene con las bases de datos). Las dependencias apuntan al model.

Lo que viene

Hoy ordenaste el código en capas y usaste arreglos de objetos. En la próxima clase das el paso natural: bases de datos, donde estrenas la capa repository y empiezas a guardar datos de verdad.
