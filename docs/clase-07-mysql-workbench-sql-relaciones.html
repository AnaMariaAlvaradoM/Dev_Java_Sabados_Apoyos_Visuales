MySQL Workbench, SQL moderno y relaciones

Comprenderás cómo persistir información real, modelar relaciones entre tablas y consultar datos de una tienda con MySQL.

01FUNDAMENTO

💾 Del objeto temporal al dato persistente

Punto de partida

Tu aplicación registró 40 pedidos. Cierras el programa, apagas el computador y vuelves al día siguiente. ¿Dónde quedaron esos pedidos?

Hasta ahora, los objetos que creabas vivían en la memoria RAM. Esa memoria es rápida, pero temporal: se limpia cuando el proceso termina.

Un negocio real necesita que clientes, pedidos, perfiles y movimientos sobrevivan al cierre del programa. Esa capacidad se llama persistencia.

Cómo imaginarlo

La RAM es como una mesa de trabajo: mientras estás allí, todo está disponible. La base de datos es el archivo permanente donde guardas lo que debe seguir existiendo mañana.

Definición técnica

La persistencia es la capacidad de conservar el estado de los datos más allá del tiempo de ejecución de una aplicación.

En palabras sencillas

Es lograr que la información no desaparezca cuando cierras el programa.

Ejemplo real

El saldo de una cuenta bancaria debe seguir igual aunque el sistema se reinicie.

Caso de negocio

Tienda digital

La tienda necesita registrar usuarios, perfiles, clientes y pedidos. Guardaremos esos datos en tablas relacionadas para consultarlos sin depender de objetos temporales.

Antes de continuar

¿Guardar la información en una lista de Java resuelve la persistencia?
Una lista funciona mientras la aplicación está ejecutándose, pero sus datos desaparecen al finalizar el proceso. Para conservarlos se necesita un mecanismo persistente, como una base de datos.
02CONCEPTO

🗄️ MySQL Server, SQL y Workbench

En esta clase trabajarás con tres piezas relacionadas, pero diferentes. Comprender su función evita uno de los errores más comunes al comenzar con bases de datos.

Definición técnica

MySQL es un sistema gestor de bases de datos relacionales que almacena información en tablas y procesa instrucciones escritas en SQL.

En palabras sencillas

Es el motor que guarda, organiza y entrega los datos cuando se los solicitas.

Ejemplo real

Funciona como un bibliotecario que sabe dónde está cada libro y responde a peticiones precisas.

Las tres piezas

MySQL Server guarda los datos. SQL es el lenguaje con el que solicitas operaciones. MySQL Workbench es la interfaz visual desde la que escribes y observas los resultados.

Buena práctica

No confundas la interfaz con el motor. Cerrar Workbench no elimina la base ni detiene necesariamente el servidor.

Antes de continuar

Si Workbench es la interfaz, ¿quién guarda realmente los datos?
El motor MySQL Server es quien almacena y procesa la información. Workbench solo facilita la comunicación con él mediante una interfaz gráfica.
03CONFIGURACIÓN

⚙️ Instalación y configuración inicial

Objetivo técnico

Preparar el entorno local

Instalarás el servidor y la herramienta visual para construir y consultar la base de datos de la tienda desde tu computador.

  1. 01

    Descarga MySQL Installer Community desde el sitio oficial de MySQL.

  2. 02

    Ejecuta el instalador y selecciona Developer Default para incluir Server y Workbench.

  3. 03

    Conserva el puerto 3306 y la configuración de desarrollo.

  4. 04

    Define una contraseña segura para el usuario administrador root y guárdala.

  5. 05

    Activa el servicio de Windows para que MySQL pueda iniciar con el sistema.

  6. 06

    Abre Workbench y prueba la conexión local con la contraseña configurada.

Error frecuente

Qué ocurre: aparece “Access denied for user 'root'”.

Cómo leerlo: MySQL está rechazando las credenciales del usuario administrador.

Cómo resolverlo: verifica la contraseña, el usuario y que la conexión apunte al servidor local correcto.

Buena práctica

Usa root solo para aprendizaje y administración local. En proyectos reales se crean usuarios con permisos limitados.

04HERRAMIENTA

🖥️ Reconocer MySQL Workbench

Tres zonas esenciales

El panel Schemas muestra las bases y tablas. El Query Editor recibe instrucciones SQL. El panel de resultados informa qué ocurrió y muestra los datos recuperados.

Navigator / Schemas

Representa la estructura disponible en el servidor conectado.

Query Editor

Es el espacio donde escribes y ejecutas instrucciones SQL.

Result Grid

Muestra en filas y columnas la información devuelta por una consulta.

Antes de continuar

Tienes cinco instrucciones escritas, pero necesitas ejecutar solo una. ¿Debes borrar las otras cuatro?
No. Puedes ubicar el cursor en la instrucción deseada y ejecutar la sentencia actual, o seleccionar únicamente el bloque que necesitas correr.
05IMPLEMENTACIÓN

🏗️ Crear la base de datos

Antes de crear tablas necesitas un contenedor lógico. En MySQL ese contenedor es la base de datos.

Qué vamos a hacer

Crearemos la base tienda y la seleccionaremos como contexto de trabajo.

Por qué lo hacemos

MySQL necesita saber dentro de qué base debe crear las tablas posteriores.

Qué aprenderás

La diferencia entre crear una base y activarla para ejecutar instrucciones.

EJECUTARMySQL Workbench · Query Editor
CREATE DATABASE tienda;
USE tienda;

Error frecuente

Qué ocurre: MySQL indica que la base ya existe.

Cómo leerlo: se intentó crear nuevamente un contenedor con el mismo nombre.

Cómo resolverlo: reutiliza la base existente o elimina únicamente una base de prueba que estés seguro de no necesitar.

06MODELADO

📊 Tablas, columnas y tipos de datos

Una tabla representa una entidad del negocio. Cada columna describe una característica y cada fila representa un registro concreto.

Puente con POO

Una clase se parece a una tabla, un atributo a una columna y un objeto a una fila.

Definición técnica

Un tipo de dato establece qué valores puede almacenar una columna y cómo debe tratarlos el motor.

En palabras sencillas

Define si una columna guarda texto, números, fechas o valores lógicos.

Ejemplo real

Un correo es texto, una fecha de registro es una fecha y el estado activo se representa como verdadero o falso.

Qué vamos a hacer

Crearemos la tabla usuario con restricciones para proteger la calidad de los datos.

Por qué lo hacemos

El negocio necesita correos únicos, nombres obligatorios y un estado inicial definido.

Qué aprenderás

Cómo combinar tipos de datos, obligatoriedad, unicidad y valores predeterminados.

CREARMySQL Workbench · Query Editor
CREATE TABLE usuario (
    id INT AUTO_INCREMENT PRIMARY KEY,
    nombre VARCHAR(100) NOT NULL,
    correo VARCHAR(150) NOT NULL UNIQUE,
    activo BOOLEAN DEFAULT TRUE,
    fecha_registro DATE
);

Qué vamos a hacer

Consultaremos la estructura interna que MySQL registró para la tabla.

Por qué lo hacemos

Necesitamos comprobar tipos, restricciones y propiedades antes de insertar información.

Qué aprenderás

A inspeccionar una tabla con DESCRIBE.

EJECUTARMySQL Workbench · Query Editor
DESCRIBE usuario;

Buena práctica

Para valores monetarios utiliza DECIMAL. Evita tipos de punto flotante cuando la exactitud financiera sea importante.

Qué vamos a hacer

Insertaremos dos usuarios y luego consultaremos todas sus columnas.

Por qué lo hacemos

Una tabla sin filas solo describe estructura; los registros permiten comprobar la persistencia.

Qué aprenderás

Cómo usar INSERT INTO, valores predeterminados y SELECT.

EJECUTARMySQL Workbench · Query Editor
INSERT INTO usuario (nombre, correo, fecha_registro)
VALUES ('Ana Torres', 'ana@correo.com', '2026-07-01');

INSERT INTO usuario (nombre, correo, fecha_registro)
VALUES ('Carlos Ruiz', 'carlos@correo.com', '2026-07-05');

SELECT * FROM usuario;
07IDENTIDAD

🔑 Identificar cada fila sin ambigüedad

Problema de negocio

Existen dos clientes llamados Carlos Ruiz. ¿Cómo sabrá el sistema a cuál pertenece un pedido?

Definición técnica

Una llave primaria identifica de forma única cada fila, no admite duplicados y no puede ser nula.

En palabras sencillas

Es el dato que permite distinguir un registro de todos los demás.

Ejemplo real

Dos personas pueden compartir nombre, pero no el mismo número de identificación.

Dos decisiones importantes

PRIMARY KEY declara la identidad de la fila. AUTO_INCREMENT permite que MySQL asigne el siguiente número automáticamente.

Buena práctica

Define una llave primaria en toda tabla. Aunque el motor permita omitirla, el modelado profesional necesita una identidad estable.

08RELACIONES

🔗 Relación uno a uno

Un usuario necesita datos esenciales para autenticarse, mientras que su perfil puede contener información adicional. Separarlos mantiene cada tabla enfocada.

Definición técnica

En una relación 1:1, una fila de una tabla se asocia con una sola fila de la otra tabla, y viceversa.

En palabras sencillas

Cada usuario tiene un perfil y cada perfil pertenece a un único usuario.

Ejemplo real

Una persona posee un expediente principal y ese expediente corresponde únicamente a esa persona.

Qué vamos a hacer

Crearemos la tabla perfil y la conectaremos con usuario.

Por qué lo hacemos

Queremos separar datos de acceso y datos complementarios sin perder la relación entre ambos.

Qué aprenderás

Cómo implementar una llave foránea y usar UNIQUE para garantizar el 1:1.

CREARMySQL Workbench · Query Editor
CREATE TABLE perfil (
    id INT AUTO_INCREMENT PRIMARY KEY,
    usuario_id INT NOT NULL UNIQUE,
    bio VARCHAR(255),
    telefono VARCHAR(20),
    FOREIGN KEY (usuario_id) REFERENCES usuario(id)
);

Qué vamos a hacer

Crearemos un perfil para cada usuario existente.

Por qué lo hacemos

Necesitamos probar que la llave foránea apunta a filas reales.

Qué aprenderás

Cómo insertar datos relacionados mediante el identificador del usuario.

EJECUTARMySQL Workbench · Query Editor
INSERT INTO perfil (usuario_id, bio, telefono)
VALUES (1, 'Desarrolladora backend', '3001234567');

INSERT INTO perfil (usuario_id, bio, telefono)
VALUES (2, 'Disenador UX', '3009876543');

SELECT * FROM perfil;

Error frecuente

Qué ocurre: MySQL rechaza un segundo perfil para el mismo usuario.

Cómo leerlo: el mensaje indica una entrada duplicada sobre la columna usuario_id.

Cómo resolverlo: modifica el perfil existente o usa otro usuario. La restricción UNIQUE está protegiendo la relación 1:1.

09RELACIONES

🧩 Relación uno a muchos

Problema de negocio

Un cliente puede comprar varias veces. ¿Debemos repetir todos sus datos dentro de cada pedido?

Repetir el nombre, la ciudad y otros datos del cliente en cada pedido produciría duplicidad y riesgo de inconsistencias.

Cómo piensa la base

El cliente se guarda una sola vez. Cada pedido conserva únicamente el identificador del cliente al que pertenece.

Definición técnica

Una relación 1:N permite que una fila de la tabla principal se asocie con varias filas de una tabla dependiente.

En palabras sencillas

Un cliente puede tener muchos pedidos, pero cada pedido pertenece a un solo cliente.

Ejemplo real

Una persona puede tener muchas facturas, pero cada factura tiene un único comprador.

Qué vamos a hacer

Crearemos las tablas cliente y pedido.

Por qué lo hacemos

El negocio necesita registrar varios pedidos por cliente sin duplicar la información del comprador.

Qué aprenderás

Cómo una llave foránea sin UNIQUE permite repetir el mismo cliente en múltiples pedidos.

CREARMySQL Workbench · Query Editor
CREATE TABLE cliente (
    id INT AUTO_INCREMENT PRIMARY KEY,
    nombre VARCHAR(100) NOT NULL,
    ciudad VARCHAR(80)
);

CREATE TABLE pedido (
    id INT AUTO_INCREMENT PRIMARY KEY,
    cliente_id INT NOT NULL,
    producto VARCHAR(120) NOT NULL,
    monto DECIMAL(10,2) NOT NULL,
    fecha DATE,
    FOREIGN KEY (cliente_id) REFERENCES cliente(id)
);

Qué vamos a hacer

Insertaremos dos clientes y tres pedidos relacionados.

Por qué lo hacemos

Necesitamos observar que el identificador del primer cliente puede aparecer en varios pedidos.

Qué aprenderás

Cómo se representa el lado “muchos” mediante valores repetidos en la llave foránea.

EJECUTARMySQL Workbench · Query Editor
INSERT INTO cliente (nombre, ciudad)
VALUES ('Laura Gomez', 'Bogota');

INSERT INTO cliente (nombre, ciudad)
VALUES ('Pedro Diaz', 'Medellin');

INSERT INTO pedido (cliente_id, producto, monto, fecha)
VALUES (1, 'Teclado mecanico', 180000.00, '2026-07-10');

INSERT INTO pedido (cliente_id, producto, monto, fecha)
VALUES (1, 'Mouse gamer', 95000.00, '2026-07-11');

INSERT INTO pedido (cliente_id, producto, monto, fecha)
VALUES (2, 'Monitor 24', 620000.00, '2026-07-12');

Error frecuente

Qué ocurre: se intenta guardar un pedido con un cliente_id que no existe.

Cómo leerlo: MySQL informa que falla una restricción de llave foránea.

Cómo resolverlo: usa el identificador de un cliente existente o crea primero el cliente correspondiente.

10CONSULTAS

🔎 Convertir datos en respuestas de negocio

Guardar datos no es suficiente. El valor aparece cuando puedes responder preguntas concretas del negocio con consultas precisas.

Qué vamos a hacer

Seleccionaremos únicamente el nombre y la ciudad de cada cliente.

Por qué lo hacemos

No siempre necesitas recuperar todas las columnas de una tabla.

Qué aprenderás

Cómo elegir columnas específicas con SELECT.

EJECUTARMySQL Workbench · Query Editor
SELECT nombre, ciudad
FROM cliente;

Qué vamos a hacer

Filtraremos los pedidos cuyo monto sea mayor a 100000.

Por qué lo hacemos

El gerente necesita analizar únicamente compras de mayor valor.

Qué aprenderás

Cómo limitar filas mediante una condición WHERE.

EJECUTARMySQL Workbench · Query Editor
SELECT producto, monto
FROM pedido
WHERE monto > 100000;

Qué vamos a hacer

Ordenaremos los pedidos desde el más costoso hasta el menos costoso.

Por qué lo hacemos

Un reporte debe presentar los datos en un orden útil para tomar decisiones.

Qué aprenderás

Cómo usar ORDER BY con orden descendente.

EJECUTARMySQL Workbench · Query Editor
SELECT producto, monto
FROM pedido
ORDER BY monto DESC;
Unir información relacionada

Qué vamos a hacer

Mostraremos el nombre del cliente junto con cada producto y monto comprado.

Por qué lo hacemos

La tabla pedido guarda el identificador del cliente, pero el reporte necesita su nombre.

Qué aprenderás

Cómo unir tablas relacionadas mediante JOIN.

EJECUTARMySQL Workbench · Query Editor
SELECT cliente.nombre, pedido.producto, pedido.monto
FROM cliente
JOIN pedido ON cliente.id = pedido.cliente_id
ORDER BY cliente.nombre;

Antes de continuar

¿Por qué no guardamos directamente el nombre del cliente dentro de cada pedido?
Repetir el nombre en cada pedido produciría duplicidad y posibles inconsistencias. La relación conserva una sola fuente de verdad para el cliente y JOIN recupera la información cuando se necesita.
